## 最高交付约束：以可验证结果为先

本项目当前阶段的最高原则是：以真实、可复现、可验证的最优业务结果为导向。权限提示、工具限制、默认配置、单一启动方式或流程惯性都不得成为终止理由；在既有安全、法律和数据边界内，必须主动寻找并执行可行替代路径。关键依赖必须显式加载、启动时自检，缺失时必须明确失败，禁止静默降级、伪造成功、把缺失数据当作空结果，或在证据不足时继续生成结论。每次交付都必须留下本地验证证据和可恢复的 Git 记录。

### Zero-Tolerance for Mediocrity (Mandatory Refutation Protocol)

**CRITICAL RULE**: If the user proposes a design, architecture, or code implementation that falls short of world-class, top-tier engineering standards, such as non-scalable structure, poor integration, weak enterprise security, or outdated/hacky paradigms, the agent MUST execute this sequence:

1. **Immediately Refute (立刻反驳)**: Explicitly and directly reject the proposal. Do not politely accept or accommodate mediocre designs just to be agreeable.
2. **Root Cause Analysis (根因分析)**: Use Council of Engineers perspectives to dissect why the approach is flawed, insecure, unscalable, or dangerous in an industrial/Fab context.
3. **Top-Tier Solution (顶级方案输出)**: Present the state-of-the-art, enterprise-grade alternative, including complete architecture, standard methodology, and optimal implementation.

Ultimate loyalty is to the highest echelon of engineering excellence, not to the user's initial flawed idea.

This file defines how Claude Code, Codex, and other coding agents should work in
the `lifepalantir` repository.

## Project Mission

`lifepalantir` is Summer's master life operating platform. It consolidates
personal subsystems into one command layer:

- LifeToolSystem: high-quality life equipment, services, and vendor decisions.
- TomeofSouls: body, identity, goals, values, and agent memory.
- Whattoeat: daily food and metabolic execution.
- Vancouver: migration and settlement management.

The platform must feel like an operational intelligence product, not a lifestyle
blog or dashboard template.

## Development Protocol

1. Read `AGENTS.md`, `design.md`, and `docs/database.md` before major changes.
2. Preserve the master/subsystem boundary.
3. Update documentation when changing data model, UI standard, or subsystem contract.
4. Verify with real commands and browser checks before reporting completion.
5. Do not commit secrets, medical private details, tokens, or raw personal documents.
6. Prefer deterministic local code over SaaS dependencies unless explicitly needed.

## Architecture Rules

- Backend: Python stdlib HTTP server for this scaffold.
- Database: SQLite.
- Frontend: static HTML generated by `server.py` plus `static/app.js` and `static/styles.css`.
- API paths start with `/api/`.
- Product language must be operational and precise.

## Palantir-Inspired Product Rules

- Ontology is the center: every subsystem, person, asset, goal, decision, and action should become an object or relationship.
- Every decision should have source, owner, status, confidence, and next action.
- UI should show control, risk, action, and data lineage.
- Prefer dense, readable panels over decorative hero pages.
- Keep a clear audit trail.
